Reflections:

The exercise was about creating an object model for a supermarket. If it was software for an online supermarket I would definitely stay with Customer Class. However, if it was a regular supermarket, a customer class would not be necessary in my opinion. I improved my class diagram by adding proper relationships, and I realized that not all variables and methods should be public.

Learning outcomes:

1) I have gain the ability to read class diagrams and analyze them to implement their structure into code/scripts. Moreover, I am able to do my own system architecture and organize my ideas in class diagrams showing proper relationships and inheritance.

2) Class diagrams are a very useful graphical representation of system design based on object oriented programming. However, before the class diagram can be made, it is important to properly choose what element of the system should be a class. The most important thing that I have learnt, is: if is it has data and behaviors, it should be a class. However, it does not mean that data without behaviour should not be represented by a class. It can, and this approach can be clearly found in Kotlin, in which Data Classes are commonly used and they are strictly dedicated for data storage:

data class User(val name: String, val age: Int)

3) There are three steps of creating a class diagram: a) Identify classes, their attributes and behaviours; b) find relationships between classes; c) Create the structure of the system based on classes and their relationships.

4) I have learnt a variety of relationships between classes that are implemented in class diagrams:
Association (a general relationship between two co-existing classes where they play some role with each other); Dependency (one class depends on another class by using its object as a variable or creating its objects); Aggregation (one class consist of objects another class); Composition (it is a stronger version of aggregation because instances of once class cannot exist without the other). And finally Inheritance (hierarchical relationships where a subclass is a more specialized type of superclass).

5) Encapsulation is a very important aspect of object-oriented programming. Private variables basically cannot be acceded form outside of a class (getters and setters are needed for that). Protected variables, however, are accessable only for the class and its subclasses.
